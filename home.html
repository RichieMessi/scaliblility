Systems Architecture Diagram

Components

Starting from left to right I will list all the component and give a brief description about each. This is not a complete solution but I believe it can solve many problems including bottlenecks during high traffic.

  • 1 - Users

    The daily users who interact with the front end of walmart's desktop and mobile application with their desktop and mobile applications

  • 2 - Frontend (The visual side of the web app to User )

    The walmart application offering multiple services and products

  • 3 - The Google Cloud Load Balancer

    This comes in handy when managing different API calls from backend API requests

  • 4 - Redis Cache (Top Right)

    This helps in reducing network traffic and provides efficient page loads

  • 5 - Databases (Bottom Right)

    This is where all the data is stored in the application

Notice how the Databases, Redis Cache and Front-end of the application are enclosed in rectangles. Those are Docker containers (the key to the scalibility of this architecture)

Moreover, we can implement our API's with newer languages such as Golang and Elm which are specifically designed for concurrency. Such languages can take full advantage of multiple cores on the machines and provide optimum performance . (Docker and Kubernetes are both written in golang)

Building the front end in React or Angular and backend in Node

React and angular are the defacto goto's (my personal opinion) for big organisations to build their web apps due to high traffic. High traffic leads to higher API calls and often requires extreme DOM manipulation. React's JSX and Angular's Typescript both handle this with ease as they follow a comopnent based approach. So choosing either one would be a good choice.

I would personally go with React as it favors functional style of programming where functions can be composed into other functions. This is the very reason why I love JavaScript.

For the backend API calls, Node is a brillliant choice as it takes power of the V8 engine written in C++ and C. Javascript has always been single threaded. But Node has the cool feature of read even print loop. Now the code can still flow downwards executing the other instructions and can wait for the pending task simultaneously. Promises and async await are great examples. Now both front end and backend can be written in JS which means theres no context switch. This leads to code reusability which is always a plus

How Docker is the key to this architecture?

Docker provides light weight images which can run as a container. One image can have multuple instances. With just a few click, I was able to spin up a docker container with Ubuntu Operating System on my Windows. With just one single command, one can run an instance of MongoDB, Angular, Node etc - create a docker compose file and in a few seconds, an entire mean app can be created and that too without installing any other software.

It does not end there. Docker solves the common problem of "it works on my machine but it does not work on yours" among developers. Docker provides an isolated environment therefore, all the developers can work in the exact same environment which leads to less bugs.

All these are my personal opinions which I would consider before attempting to scale an application. There is a lot more to it and I am still learning

Add Microsoft Azure

Microsoft Azure is a great choice for developers like myself who are still learning about dev ops. Azure provides a clean user interface (all written in typescript) and everything gets done easy and quick. With a few commands, I deployed a docker container for my question 1 in azure. Great expreience. Adding multiple deployment slots for staging of an app to test a feature before release is very easy. It picks up a project right from my github and upon some configurations and scripting, it can be linked to the dockerhub aswell. So whenever a container is updated, Azure picks up the change. There are app services that can help scale individual parts of an application both horizontally and vertically.
